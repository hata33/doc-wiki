## 项目背景

### 问题：传统 RAG 的局限性

当 AI 编程助手需要使用 LangChain 等 API 时，传统 RAG（检索增强生成）存在以下问题：

1. **知识过时** - AI 训练数据中的 API 可能已废弃
2. **版本混乱** - 无法区分不同版本的 API 行为
3. **答案碎片化** - 检索结果是零散的文档片段
4. **缺乏可运行示例** - 检索到的代码可能无法直接运行

### 解决方案：LLM Wiki

将 LangChain 官方文档"编译"成 AI 可读的结构化 Wiki，优先于 RAG 使用。

---

## 核心目标

### 1. 100% 覆盖率
- 索引所有 langchain、langgraph、python 目录下的文档
- 确保每个原始文档都有对应的 Wiki 页面

### 2. 结构化组织
- **API 页面** (`wiki/api/`) - 聊天模型、嵌入、向量存储等
- **概念页面** (`wiki/concepts/`) - LangChain、LangGraph 核心概念
- **源文档索引** (`wiki/sources/`) - 所有 Python 集成

### 3. 可追溯性
- 每个 Wiki 页面包含原始文档路径
- 原始文档链接可直接点击跳转
- 使用 `[[PageName]]` 格式建立交叉引用

### 4. 版本感知
- 明确标注 LangChain 1.0.x、LangGraph 1.0.x 版本
- 标注 API 状态（当前/废弃/实验性）

## 构建思路

### 第一阶段：扫描和分类

**输入**: `raw/src/oss/` 目录下的所有 `.mdx` 文件

**处理流程**:
```python
1. 扫描三个目录：
   - langchain/     (71 个文件)
   - langgraph/     (38 个文件)
   - python/        (1,345 个文件)

2. 根据路径自动分类：
   - python/integrations/chat/      → api/
   - python/integrations/embeddings/ → api/
   - python/integrations/vectorstores/ → api/
   - langchain/                      → concepts/
   - langgraph/                      → concepts/
   - 其他                            → sources/
```

### 第二阶段：生成 Wiki 页面

**命名规则**（避免冲突）:
```python
# 使用完整相对路径作为 Wiki 名称
python/integrations/chat/openai.mdx
→ python_integrations_chat_openai

langchain/agents.mdx
→ langchain_agents
```

**页面结构**:
```markdown
---
name: page_name
category: api|concepts|sources
source: relative/path/to/doc.mdx
---

## 描述
标题 - 描述

## 原始文档
- **主文档**: [`path`](../../raw/path)

## 文档信息
- **标题**: xxx
- **描述**: xxx
- **源路径**: `xxx`

## 说明
这是原始文档的索引页面。点击上方链接查看完整文档内容。

## 相关页面
- 查看原始文档获取完整信息、示例和配置选项
```

### 第三阶段：生成分类索引

**目的**: 为 Claude Code 提供完整的文档列表

**文件**:
- `wiki/index/api_index.md` - 406 个 API 完整列表
- `wiki/index/concepts_index.md` - 363 个概念完整列表
- `wiki/index/sources_index.md` - 1,751 个源文档完整列表

**分类方式**:
```python
# API 索引按类型分组
- 聊天模型 (102 个)

- 向量存储 (7 个)
- 检索器 (72 个)
- 文本分割 (1 个)
- 链和解析 (3 个)
- 记忆 (4 个)
- 其他 (125 个)

# 源文档索引按集成类型分组
- 聊天模型集成 (8 个)
- 文档加载器 (199 个)
- 工具集成 (160 个)
- LLM 集成 (97 个)
- 其他 (1,261 个)
```

### 第四阶段：验证和修复

**验证内容**:
```python
1. 检查所有 Wiki 页面的 source 路径
2. 验证原始文档是否存在
3. 修复路径格式问题
   - 移除多余的 raw/ 前缀
   - 统一路径格式
4. 确保所有链接有效
```

**结果**: 2,520/2,520 链接有效 (100%)

```

5. Read(../../raw/python/integrations/chat/openai.mdx)
6. 综合给出答案（含代码、版本、示例）
```

**用户问题**: "如何集成 AWS Bedrock？"

**用户问题**: "有哪些可用的向量存储？"

### 三层查找策略

**第一层：主索引** (`wiki/index/index.md`)

**第二层：分类索引** (`wiki/index/*_index.md`)

## 技术架构

### 目录结构

```
langchain/
├── 使用指南.md                # 用户指南
├── claude_prompt_template.md # 提示词模板
├── .gitignore                 # 排除 raw/ 目录
├── raw/                       # LangChain 官方文档（独立 Git 仓库）
├── wiki/                      # 生成的 Wiki
│   ├── index/
│   │   ├── index.md          # 主索引
│   │   ├── api_index.md      # API 完整索引
│   │   ├── concepts_index.md # 概念完整索引
│   │   └── sources_index.md  # 源文档完整索引
│   ├── api/                  # API 页面 (406 个)
│   ├── concepts/             # 概念页面 (363 个)
│   └── sources/              # 源文档索引 (1,751 个)
└── schema/                    # 构建脚本
    └── scripts/
        ├── build_wiki.py     # 构建 Wiki
        ├── verify_wiki.py    # 验证和修复
        └── README.md         # 脚本说明
```

### 核心脚本

**1. build_wiki.py** (262 行)
```python
# 主要功能
- get_wiki_name()        # 生成唯一 Wiki 名称
- determine_category()   # 确定文档分类
- create_wiki_content()  # 创建 Wiki 内容
- build_docs_index()     # 构建文档索引
- build_category_indexes() # 构建分类索引

# 使用方式
python build_wiki.py              # 完整构建
python build_wiki.py --index-only # 只生成索引
python build_wiki.py --category   # 只生成分类索引
```

**2. verify_wiki.py** (182 行)
```python
# 主要功能
- verify_links()      # 验证链接有效性
- fix_source_links()  # 修复路径格式
- show_stats()        # 显示统计信息

# 使用方式
python verify_wiki.py              # 验证链接
python verify_wiki.py --fix        # 验证并修复
python verify_wiki.py --stats      # 显示统计
```

## 设计决策

### 1. 为什么不用 LLM 生成内容？

**决策**: 直接索引原始文档，不用 LLM 重写

**原因**:
- ✅ 忠于原始文档，避免幻觉
- ✅ 可自动更新，无需人工审核
- ✅ 文件大小可控（每个 ~20 行）
- ✅ 链接可点击，便于验证

### 2. 为什么使用完整路径作为 Wiki 名称？

**决策**: `python_integrations_chat_openai` 而非 `ChatOpenAI`

**原因**:
- ✅ 避免命名冲突（1,454 个文件）
- ✅ 名称唯一且可预测
- ✅ 便于通过文件名查找

### 3. 为什么创建分类索引？

**决策**: 除了主索引，还创建 api/concepts/sources 索引

**原因**:
- ✅ Claude Code 可以快速浏览完整列表
- ✅ 不需要在主索引中列出 1,751 个文件
- ✅ 支持探索性查询

### 4. 为什么排除 raw/ 目录？

**决策**: raw/ 作为独立的 Git 仓库，不提交到 Wiki 仓库

**原因**:
- ✅ 避免仓库过大（raw/ 有数千个文件）
- ✅ 可以独立更新官方文档
- ✅ Wiki 仓库更轻量，便于克隆
